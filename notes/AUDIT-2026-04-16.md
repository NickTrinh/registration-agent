## Findings fixed this session

### 🔴 F-01 — What-If tool loses `studentId` on service-worker unload

**Where:** [service-worker.ts:65-75](../src/background/service-worker.ts)

**Problem:** `cachedStudentId` and `cachedStudentGoal` were module-level variables. Chrome MV3 service workers unload after ~30 seconds of idle, wiping module state. The What-If tool failed with "No audit loaded yet" whenever the student opened the extension after a pause — even though the audit was already in storage.

**Fix:** Introduced `persistStudentCache()` + `hydrateStudentCache()` helpers that mirror the cache to `chrome.storage.local`. `executeWhatIf` now re-hydrates on each call.

**Broader lesson:** Any module-level state the worker depends on across events MUST be backed by `chrome.storage.local`. Already applied to `memories`, `provisional`, `curator_turns`, `auditText`, `profile`. Now also `studentId` + `studentGoal`.

### 🔴 F-02 — `addMemory` has no deduplication

**Where:** [memory-store.ts:54-78](../src/background/agent/memory-store.ts)

**Problem:** Onboarding saves 5-10 memories per intake. Sonnet's natural tendency is to save multiple near-duplicate entries ("theology classes", "theology interest", "Middle Eastern theology"). The store appended all of them. Student saw a cluttered memory panel and the routing index grew with redundant lines.

**Fix:** Added `normalizeForDedup()` (lowercase, strip punctuation) + `jaccardSim()` (word-set overlap) helpers. Before appending, `addMemory` now checks for a same-type entry with either (a) exact normalized match or (b) Jaccard similarity ≥ 0.6. On match, merges content (keeping the longer variant) and bumps `lastAccessedAt` instead of appending.

**Threshold rationale:** 0.6 is aggressive enough to catch "theology classes" / "theology courses" (shared: theology; different: classes/courses → ~0.33) vs "theology classes" / "theology middle east" (shared: theology; different: classes/middle/east → ~0.25). May need tuning after more real use. Stub threshold constant (`DEDUP_SIMILARITY_THRESHOLD`) exposed for easy adjustment.

### 🔴 F-03 — Markdown tables don't render

**Where:** [AuditChat.tsx](../src/sidebar/pages/AuditChat.tsx)

**Problem:** `react-markdown` by default doesn't support GFM tables. When Sonnet emitted `| col1 | col2 |` syntax (which it does for requirements lists, credit checks, etc.), the table rendered as raw pipes inline. This was the "formatting issues" complaint.

**Fix:** Installed `remark-gfm`, wired as `remarkPlugins={[remarkGfm]}`. Also added component overrides for `table`, `thead`, `tbody`, `tr`, `th`, `td`, `hr`, `a`, `blockquote`, `pre`, `em` (previously missing). Bumped paragraph line height and list padding for general readability.

### 🟡 F-04 — Onboarding completion doesn't clear chat

**Where:** [AuditChat.tsx:150-163](../src/sidebar/pages/AuditChat.tsx)

**Problem:** When Sonnet emitted `**Onboarding complete.**`, the mode flag flipped correctly but the chat history stayed visible. Student was confused — are they still in intake? Is the next question a follow-up?

**Fix:** After marker detection, set a 4-second `setTimeout` to clear messages + session storage. Student has time to read the summary, then the chat resets with a clean canvas. On next message, they're unambiguously in normal mode.

**Alternative considered:** Keep the summary visible but add a divider + "New conversation" button. More explicit but more UI surface. Deferred — clearing is simpler and matches how Claude Code presents transitions.

### 🟡 F-05 — Onboarding prompt encourages over-saving

**Where:** [service-worker.ts:448](../src/background/service-worker.ts) (ONBOARDING_SYSTEM_PROMPT)

**Problem:** The prompt said "Each time the student states a durable fact... call save_memory IMMEDIATELY. Don't batch. Don't wait." This invited one-save-per-sentence behavior, which compounded with F-02 to produce the over-saturation symptom.

**Fix:** Rewrote the "Saving memories" section to instruct **"one memory per distinct topic, not one per statement"**, target 5-8 high-quality entries by end-of-intake, and explicitly ask the two check questions ("Did I already save something about this topic?" / "Is this a restatement?") before each call.

**Belt-and-suspenders:** F-02 (store-level dedup) catches what the prompt misses. Both together should eliminate the symptom.

### 🟡 F-06 — Banner ID logged to DevTools console

**Where:** [service-worker.ts:283](../src/background/service-worker.ts) (before fix)

**Problem:** `console.log(... Fetching audit for student ${student.id} ...)` dumped the Banner ID to the DevTools console. Not sent to Anthropic, but appeared in screenshots if the student opened DevTools for debugging. Technical violation of ADR 0009's "never log full audit response bodies — log lengths and IDs only" rule since Banner ID is an identifier.

**Fix:** Replaced with `console.log("[FordhamHelper] Fetching audit (school/degree, catalogYear)")`. School/degree/catalog year are not PII.

### 🔵 F-07 — Markdown components were minimal

**Where:** [AuditChat.tsx:603-613](../src/sidebar/pages/AuditChat.tsx) (before fix)

**Problem:** Only `p`, `strong`, `ul`, `ol`, `li`, `code`, `h1-3` were styled. Everything else (links, horizontal rules, blockquotes, code blocks, italic) fell back to browser defaults with no Tailwind styling, making longer responses look inconsistent.

**Fix:** Added all missing components with Fordham-maroon theming and consistent spacing. See F-03 diff.

## Surfaced for discussion (not yet fixed)

### 🟡 D-01 — Live "thinking/status" tracker

**Testing issue:** #2 in [fixFordhamHelper.md](../../fixFordhamHelper.md)

**What exists:** A generic spinner appears under the message input while Sonnet is responding. Tool chips (🔍🧠💾🗑️🔮) render when tools fire, but only AFTER the tool call completes (when AI_TOOL_USE arrives). During the "planning" phase — after the user hits send but before Sonnet decides on a tool or starts streaming — the student sees nothing but the spinner.

**What's missing:** Claude-Code-style status updates: "Thinking...", "Checking your audit...", "Searching CISC electives...", "Recalling memory #3..." as the model reasons through the turn.

**Three implementation options:**

**Option A — Stream prefix detection.** When Sonnet starts streaming, check if the first ~50 chars match a known pattern ("Let me search...", "Looking at your audit..."). Render a synthetic status pill. Simple but brittle.

**Option B — Dedicated status prompt.** Add a system-prompt instruction: "Before calling any tool, emit a single line starting with `STATUS:` describing what you're about to do. The line is consumed by the UI and not shown to the student." Parser strips STATUS lines from the rendered text. More reliable; mild prompt engineering risk (Sonnet might skip the status line sometimes).

**Option C — Tool-call preview via SSE.** When `content_block_start` fires for a tool_use block, we can see the tool name before the input is fully streamed. Render a pre-chip: "🔍 Preparing search..." that fills in with the input once complete. Minimal prompt changes; uses the SSE event stream we're already consuming.

**Recommendation:** Option C is the most natural fit given the existing chip architecture. Effort: 20-30 min. Would also eliminate the "empty assistant bubble" race where chips render before any text streams.

### 🟡 D-02 — DegreeWorks session expiry handling

**Not tested yet, but will happen:** Fordham's DegreeWorks cookies refresh as long as the student touched the site within ~1 hour. After a longer idle, `fetchStudentMyself()` will 401 and the whole chain breaks.

**Current behavior:** Red `AUDIT_ERROR` banner with whatever error message came back. Student might not know what to do.

**Proposed fix:** Detect 401/403 from DegreeWorks specifically, show a targeted CTA banner: *"Your DegreeWorks session expired. Open [degreeworks](https://dw-prod.ec.fordham.edu/...) and log in again, then come back here."* Link opens in a new tab. On the next audit refresh, the red banner clears automatically.

**Effort:** 20 min. Not urgent for a demo where the user just logged in. Required for a submission-polish pass.

### 🔵 D-03 — Memory consolidation pass (Haiku-powered)

**What exists:** F-02 does exact + Jaccard dedup at add-time. Works for simple cases.

**What doesn't exist:** A periodic consolidation pass that reviews all memories, identifies overlapping clusters, and rewrites them into single canonical entries. This is what MemGPT / A-MEM call "compaction."

**Why you might want this:** After a few weeks of real use, the store will have memories that aren't exact duplicates but are semantically overlapping. Jaccard won't catch "committed to computational neuroscience grad school" + "interested in PhD programs in comp neuro" — different words, same fact.

**Implementation sketch:** A new Haiku call triggered manually (Settings button "Consolidate memories") or automatically on a count threshold (e.g. when store hits 30 entries). Haiku reads the full store, proposes merges/deletions, user reviews in a modal, applies.

**Effort:** 1-2 hours. Genuinely useful but pure polish for the sprint. Post-demo win.

### 🔵 D-04 — Prompt injection risk from audit content

**Observation:** The audit is fetched from a trusted Fordham source, BUT the audit includes free-text fields that pass through unchanged — course titles ("Intro to Philosophy"), block descriptions, `beginClass`/`endClass` strings. If a malicious actor could modify those fields (they can't — Fordham controls DegreeWorks), they could inject prompt text into our system prompt.

**Is this a real concern?** No, because the attacker would need write access to DegreeWorks. But it's worth noting in the architecture doc because:
- It documents the trust boundary explicitly (Fordham = trusted)
- If we ever ingest external data (e.g. Professor ratings, student-submitted notes), we'd need to sanitize

**Effort:** 10 min of documentation in ADR 0009 or a new ADR. No code change.
