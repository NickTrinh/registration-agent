# 0010 — Prompt caching at the system-prompt breakpoint

- **Status**: Accepted
- **Date**: 2026-04-15
- **Session**: [`_ops/journal/2026-04-15.md`](../../../../_ops/journal/2026-04-15.md)
- **Related**: runs under 0003 (worker owns the Anthropic call site); downstream of 0012 (memory index is part of the cached prefix)

## Context

Each chat turn sends a large stable prefix to Anthropic:

- Instructions (~500 tokens)
- Student profile (~500 tokens)
- Memory index (0–750 tokens depending on memory count)
- Live DegreeWorks audit (~2000–3000 tokens)
- **Total: ~3000–4000 tokens of system prompt**, plus a growing tail of conversation history and tool results

The prefix doesn't change within a chat session. Turn 1 sends it once; turn 2 sends it again; turn 3 sends it again. Naively, every turn pays the full input-token cost of the prefix — both in money and in first-token latency (the server re-tokenizes the same bytes on every call).

Anthropic ships a prompt-caching feature: mark a breakpoint in the input with `cache_control: { type: "ephemeral" }`, and the server caches the tokenized prefix for 5 minutes. Subsequent requests within the TTL read the prefix from cache, paying ~10% of the input-token cost for the cached segment. Turn 2+ within a session becomes dramatically cheaper.

The question was simply: use it or not, and where to place the breakpoint.

## Decision

Wrap the entire `systemText` string in a single cache breakpoint:

```ts
const system: Anthropic.Messages.TextBlockParam[] = [
  { type: "text", text: systemText, cache_control: { type: "ephemeral" } },
];
```

One breakpoint covering the full 3–4k-token system prompt. Turn 2+ in the same 5-minute window reads the whole prefix from cache. Conversation history is NOT cached — it grows each turn and would force cache rewrites anyway.

## Alternatives considered

### Alternative A: No caching

The simplest option. Rejected because the competition provides API credits, but burning them to re-tokenize the same 3k-token prefix every turn is waste with no upside. Even with free credits, the first-token latency reduction matters for demo pacing — a ~200ms improvement per turn is visible to a judge watching the chat flow, and cached turns feel instantaneous by comparison.

### Alternative B: Cache only the audit (narrower breakpoint)

Place the cache breakpoint after the static instructions but before the profile and memory index, so only the audit-text segment is cached. Rejected because the instructions, profile, and memory index are ALSO stable turn-to-turn within a session, and there's no cost to caching a larger prefix. One breakpoint covering the full system prompt is simpler and captures 100% of the stable bytes instead of 60%.

### Alternative C: Cache the full system prompt AND the conversation history

Place a second cache breakpoint at the end of the conversation history. Rejected because conversation history grows each turn — every new user message invalidates the history cache and forces a full rewrite. The write penalty (~1x input cost) outweighs the read savings on a growing tail. Caching is valuable for STABLE prefixes, not growing ones.

### Alternative D: Client-side prompt caching

Serialize `systemText` once per session and reuse the string in memory. Rejected because this solves a problem we don't have. Client-side tokenization is fast and cheap; the actual cost is server-side tokenization and input-token billing, both of which only Anthropic's cache addresses.

## Consequences

**~90% input-token savings on turn 2+** within the 5-minute TTL window. For a typical session (5–10 turns within 5 minutes), this is a ~70–80% reduction in average per-turn input cost across the session. Real dollar cost drops from "noticeable fraction of credit budget" to "rounding error."

**First-token latency improves noticeably on cached turns**. Cache reads skip re-tokenization, so the time-to-first-streaming-chunk drops by ~150–300ms on cached calls. Invisible to grep, obvious to anyone watching the demo.

**Cache invalidates whenever `systemText` changes**. Three things can change it:

1. A new audit refresh writes a different `auditText` (rare — once per session typically, when the student revisits DegreeWorks).
2. The profile is edited in Settings or regenerated by Haiku (rare — manual action or once per fresh audit).
3. The background curator (ADR 0011) writes a new memory, changing the memory index rendered in the system prompt (most common — happens on any turn where the curator flags something).

Expected cache-hit rate: 70–80% on turns within a session, depending on curator write frequency.

**Mitigation for curator-triggered invalidations**: in practice, most chat turns are "nothing worth saving" (the curator log confirms this during stub-mode testing). The 5-minute TTL absorbs the rare mutation — we pay one cache-miss penalty (a full-length write) and then amortize across the rest of the session. The worst-case cost is one uncached turn per memory write, which is negligible at Haiku's per-turn extraction rate.

**No code complexity added**: a single-line change, `system: systemText` → `system: [{ type: "text", text: systemText, cache_control: { type: "ephemeral" } }]`. The Anthropic SDK handles cache key computation server-side; we just mark the breakpoint.

**Free to leave in after competition**: prompt caching is a production-quality optimization wanted in any real deployment. The one-line footprint means no tech debt cost to keeping it on permanently, and removing it would be a regression.

## Revisit if...

- Curator write frequency increases to the point where cache-miss rate exceeds 50% of turns (implies the curator's extraction prompt is too permissive — tune the prompt, not the caching).
- Anthropic changes the TTL or pricing model in a way that makes the single-breakpoint strategy suboptimal.
- Memory index grows past ~2k tokens, at which point separating the memory-index segment into its own breakpoint (so curator writes don't invalidate the audit cache) becomes worth the complexity.
